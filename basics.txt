git is a database

Commit:
a fundamental unit of that db is a commit
a commit is a snapshot of the entire project at one moment in time

every commit contains 3 things:
snapshot
metadata: who committed it, when and a message
parent: a pointer to the parent commit, the commit that cbefore

all commits point to their parent commits
commits always point backwards
when merge happens, the commit has 2 parents
the structure creates a DAG
a parent can have multiple child nodes created via different branches that are then merged

Branch:
a pointer; a small text file that contains a hash of a commit
"this branch points to commit 74w82hw"
branches dont contain commits, they point at them
when a new commit is made, the pointer moves forward

Head:
Pointer that tracks your location
doesnt point at a commit, points at a branch
$ git checkout feature: head moves to point at the feature branch, the one ur workng on

if a specific commit is checked out by u, head points directy to that commit
it is called the detached head state
the concsequent commits are orphaned
a commit will be garbage collected if it is not linked to a branch

3 areas where code can live
working directly
staging area
repo

checkout
$ git checkout main: head points to main
$ git checkout c1: head points to c1
no branches move
no commits change
only history is being viewed

reset 
reset moves a branch
$ git reset c1: move main to point at this commit
the ahead commits still exist but are orphaned, no branch point to them
3 modes:
--soft: moves branch only, keeps staging and working directly
multiple commits were made that need to be combined into 1

--mixed: Moves branch, clears staging, keeps working directory
changes exist in files just unstaged
u commited smth but want to restage it differently eg split it into mult commits

--hard: moves branch, clears staging AND working directory
files change
uncommited changes are erased
when work wants to be abandoned to start fresh


revert:
creates a new commit that undoes the previous commit
when u need to undo somhing that is alr pushed and shared

rebase:
when a parent node has multiple child nodes, the can either be merged
or they can be rebased
it looks at the commit made, calc changes and creats a new commit with the prev
commit as its parent insead of the base, the older org ones are garbage collected
only prefered over commits that arent pushed/ shared
