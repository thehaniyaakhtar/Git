Git is a content-addressable database
everything is stored as objects identified by a hash (based on content, not filename)
same content → same hash → no duplication
this is why git is fast and space-efficient
the fundamental unit of that db is a commit

Commit
a commit is a snapshot of the entire project at one moment in time
git does NOT literally copy every file every time
it reuses unchanged data internally (via trees/blobs)

every commit contains:
snapshot
full state of all tracked files (tree objects)

metadata
author
timestamp
commit message
parent(s)

pointer to previous commit
normal commit → 1 parent
merge commit → 2 parents

commits always point backwards
this creates a DAG

directed → commits point to parents
acyclic → no loops possible
graph → branching history
a parent can have multiple children → happens when branches diverge

Branch
a branch is just a pointer to a commit

internally: stores a commit hash
example:
main → a1b2c3d

branches do NOT contain commits
they are just labels for latest commit in that line

when you commit:
git creates a new commit
moves the branch pointer forward

HEAD
HEAD tracks your current location
usually:
HEAD → branch → commit

$ git checkout feature

HEAD now points to feature
detached HEAD:
$ git checkout <commit-id>

HEAD points directly to commit
new commits are not attached to any branch
can be garbage collected

use:
exploring old history
testing temporarily

3 Areas of Git

working directory → actual files
staging area (index) → next commit
repository (.git) → permanent history

flow:
working → staging → repo

Checkout
$ git checkout main
$ git checkout <commit-id>

moves HEAD
updates working directory
does NOT change history

Reset
$ git reset <commit-id>
moves branch pointer
commits ahead become orphaned

modes
--soft
moves branch only
keeps staging + working

--mixed (default)
moves branch
clears staging
keeps working

--hard
moves branch
clears staging + working

Revert
$ git revert <commit-id>
creates a new commit that reverses changes
does NOT delete history
use when working in shared repos

Merge vs Rebase
merge:
$ git merge <branch>
joins histories
creates merge commit
non-linear history

rebase:
$ git rebase main

replays commits on new base
linear history
rewrites commit hashes

rule:
do NOT rebase shared commits

Reflog
$ git reflog

tracks all HEAD movements
includes commits, resets, checkouts

recovery:
$ git reset --hard HEAD@{1}

Remotes
a remote connects local repo to a hosted repo on GitHub
default name: origin

$ git remote add origin <repo-url>
$ git remote -v
$ git remote remove origin
$ git remote set-url origin <new-url>
Clone
$ git clone <repo-url>

downloads full repo
sets remote automatically

Fetch vs Pull
$ git fetch

downloads changes (no merge)
$ git pull
fetch + merge

Push
$ git push

first push:
$ git push -u origin main

sets upstream branch
after that:
$ git push
Upstream Branch

default remote branch your branch tracks

$ git push -u origin main
$ git branch -vv
Merge Abort
$ git merge --abort

cancels ongoing merge
